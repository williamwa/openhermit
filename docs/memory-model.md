# OpenHermit Memory Model

This document defines the target memory model after the internal/external state split.

## Core Principle

OpenHermit distinguishes between:

- `system memory`
- `user-authored knowledge`

System memory is runtime-owned internal state.  
User-authored knowledge is external state that the agent can read, search, and edit like normal files.

They should not be conflated.

## Memory Layers

OpenHermit uses four logical layers:

1. session log
2. episodic memory
3. working memory
4. long-term memory

## 1. Session Log

Purpose:

- factual source of truth for one session
- replay
- audit
- history API
- summarization input

Storage:

- internal state
- persisted in `~/.openhermit/{agent-id}/state.sqlite`

Contents:

- user messages
- assistant messages
- tool events
- approvals
- errors
- other per-session events

Rule:

- if it happened in the session, it belongs here first

## 2. Episodic Memory

Purpose:

- compact checkpoint summaries with future retrieval value
- bridge from raw session history into higher-level memory

Storage:

- internal state
- persisted in `~/.openhermit/{agent-id}/state.sqlite`

Rule:

- episodic memory is not a raw mirror of session events
- it only stores summaries/checkpoints

Current triggers:

- explicit checkpoint
- new session switch from an adapter
- configured turn interval
- idle timeout

Suggested fields:

- `session_id`
- `ts`
- `checkpoint_type`
- `reason`
- `history_from`
- `history_to`
- `turn_count`
- `summary`

## 3. Working Memory

Working memory should be split into two layers:

- session-local working memory
- global working memory

### Session-Local Working Memory

Purpose:

- preserve the active thread state of one session
- keep continuity without replaying full history

Examples:

- current objective
- recent conclusions
- open questions
- next steps
- session-specific constraints

Storage:

- internal state
- should live in `state.sqlite`

Update strategy:

- triggered by program lifecycle after checkpoints
- generated by the agent
- built from:
  - previous session-local working memory
  - new episodic checkpoint summaries
  - current session context

### Global Working Memory

Purpose:

- hold cross-session, high-signal current context for one agent

Examples:

- important active threads
- recent cross-session decisions
- important currently running services
- short-lived cross-session context

Storage:

- internal state
- should live in `state.sqlite`

Update strategy:

- triggered by program lifecycle
- generated by the agent
- built from recent session-local working memory and recent episodic checkpoints

## 4. Long-Term Memory

Long-term memory should be split into:

- `system long-term memory`
- `user-authored knowledge`

### System Long-Term Memory

Purpose:

- stable reusable facts the runtime decides are worth preserving

Examples:

- persistent user preferences
- stable project conventions
- long-lived operational knowledge

Storage:

- internal state
- should live in `state.sqlite`

Access:

- retrieved by program-managed lookup
- not blindly injected in full

### User-Authored Knowledge

Purpose:

- files the user wants the agent to read and work with directly

Examples:

- docs
- project notes
- prompts
- curated knowledge files

Storage:

- external workspace

Access:

- `file_search`
- `read_file`
- `list_files`

## Update Responsibility

Memory updates should be split between:

- program-driven orchestration
- agent-generated content

### Program-Driven

The program should decide:

- when to checkpoint
- what transcript range to summarize
- when to refresh working memory
- when to run long-term promotion
- where each result is stored

### Agent-Generated

The agent should generate:

- episodic summaries
- rewritten working memory
- long-term promoted content

This keeps memory behavior predictable while still using the model for summarization quality.

## Current Migration Direction

Already migrated:

- session history
- session metadata
- episodic checkpoints

Still to migrate:

- session-local working memory
- global working memory
- system long-term memory

## Retrieval Strategy

### Use session log when:

- replaying history
- auditing behavior
- serving session history APIs

### Use episodic memory when:

- recalling what happened recently across checkpoints
- building or refreshing working memory

### Use working memory when:

- injecting compact active context into the prompt

### Use user-authored knowledge when:

- the agent should search, read, and manipulate external knowledge files as part of ordinary tasks

## Summary

- session log = raw facts
- episodic memory = checkpoint summaries
- working memory = active state
- long-term memory = stable system knowledge
- user-authored knowledge = external files, not system memory
